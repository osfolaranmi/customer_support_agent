## Final Reflection

1. Which part of the graph was hardest to design?

The most challenging part of the graph was designing the review and regeneration workflow. 
Instead of returning the first response generated by the LLM, I introduced a review node that evaluates the response for accuracy, completeness, groundedness, and professionalism. If the response does not meet the required quality threshold, it is automatically routed to a regeneration node before being reviewed again. Designing this feedback loop required careful consideration to avoid unnecessary regeneration while still improving response quality. It transformed the workflow from a simple linear pipeline into a more robust agentic process.


---

2. How did you decide when to use retrieval versus troubleshooting versus escalation versus onboarding?

I implemented a request classification node as the entry point of the workflow. Every customer question is first categorized into one of five routes:

- Knowledge: questions about product features, pricing, subscription plans, documentation, or policies. These queries         require Retrieval-Augmented Generation (RAG) to retrieve relevant documents before generating a response.
- Troubleshooting: issues where a customer reports that a feature is not working correctly. These requests are routed         through a troubleshooting node before retrieving supporting documentation.
- Escalation: urgent or sensitive situations such as security concerns, payment disputes, refund requests, or account         lockouts. These requests are marked with higher priority and routed to the appropriate support team.
- Onboarding: implementation, migration, setup, and training questions. These follow a dedicated onboarding route before      retrieving onboarding documentation.
- General: greetings and casual conversations that do not require specialized routing.

Using a classifier ensures that each request follows a workflow appropriate to its intent instead of treating every question identically.


---


3. What did your review node catch or improve?

The review node served as an automated quality assurance step. It evaluated every generated response against four criteria:

-  Accuracy
-  Completeness
-  Groundedness
-  Professionalism

It helped identify responses that were incomplete, insufficiently grounded in the retrieved documentation, or lacking enough detail. When a response failed the review, the workflow regenerated the answer before presenting it to the user. This additional validation step improved consistency and reduced the likelihood of returning weak or unsupported responses.


---

4. How does memory improve the customer support experience?

Memory enables the assistant to provide more personalized and context-aware support.

I implemented two types of memory:

-  Conversation memory, which preserves context within the current chat session. This allows customers to ask follow-up         questions without repeating previous information.
-  Customer memory, which stores useful customer-specific information across conversations, such as previous interactions       or preferences.

By combining these two forms of memory, the assistant can maintain continuity across both a single conversation and multiple customer sessions, creating a more natural support experience.

5. What would you change before deploying this to real customers?

Before deploying this solution in a production environment, I would implement several additional capabilities:

-  Integrate with a real CRM and ticketing system such as Salesforce, Zendesk, or Microsoft Dynamics.
-  Add authentication and authorization to ensure customer data is securely accessed.
-  Add monitoring, logging, and analytics to track response quality, latency, and customer satisfaction.
-  Improve the retrieval pipeline by introducing document reranking and semantic search optimization.
-  Expand the knowledge base and implement continuous updates as product documentation changes.


These improvements would make the solution more secure, scalable, and suitable for production use while preserving the agentic workflow developed during this project.
